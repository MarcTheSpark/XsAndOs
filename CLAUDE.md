# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A SuperCollider live-performance tool for the composition *Xs and Os* (Splice 2026). It loads MIDI "cues" and plays them back through a stand-in synth under a **hand-authored tempo envelope**, deliberately ignoring the tempo information baked into the MIDI files.

Directory layout: **`midi_cue_player.scd`** (the entry point) sits at the repo root; everything else is foldered — **`lib/`** the non-main `.scd` files, **`data/`** the plain-text config, **`midi/`** the `.mid` source files, **`plan/`** the spec/TODO docs.

Files, split for a **minimal performer interface**:
- **`midi_cue_player.scd`** (root) — the full entry point. A one-line SETUP (`this.executeFile("lib/setup.scd".resolveRelative)`, run once) followed by documented one-liners: `~play.(n)`, `~next.()`, `~stop.()`, `~tempoMul`, `~tempoGui.()`, `~boardGui`, MIDI learn, list `~cues`.
- **`midi_cue_player_simple.scd`** (root) — a stripped-down control sheet to hand to a performer: the same one-line SETUP plus only `~next`/`~stop`/`~play`, `~tempoMul`, and MIDI learn. Nothing else for them to deal with.
- **`lib/setup.scd`** — the one-shot session setup the two entry files `executeFile`: it `s.waitForBoot`s, loads the engine + GUIs, the track map, the cue sheet, the saved curves, and MIDI, then opens the board. Paths are resolved relative to `lib/` and **up-front (synchronously)** so they still work inside the async `waitForBoot` callback. A single compilation unit.
- **`lib/cuePlayerEngine.scd`** — the "library": SynthDef, state, and all `~` functions. Loaded once by `lib/setup.scd` via `this.executeFile(...)`. **It must stay a single compilation unit** (one wrapping `( … )`), because `executeFile` interprets the whole file at once — splitting it into multiple top-level `( … )` regions like the old version would be a syntax error.
- **`lib/tempoEnvGui.scd`** — the **curve editor** (built on core `EnvelopeView`). Also a single compilation unit, also `executeFile`'d by SETUP. Defines `~openTempoEnvGui` / `~tempoGui`. Edits the cue's tempo (BPM) curve **or**, via the track/param selectors, a per-`(cue, track, param)` parameter curve (the params in `~paramList`: `cutoff`/`pan`/`amp`/`chaos`/`sharp`/`sat`/`rel`). The track selector lists **only the audio tracks declared in `tracks.txt`, shown by their `name`** (plus `"all"` = the `"*"` all-tracks curve); the selected label maps back to the track-index key via `trackKeys`/`trackLabels`. The cue's MIDI notes are drawn as a faint **grey piano-roll backdrop** (x = beat over the cue length, y = pitch high-at-top) so you can shape curves against the actual notes: this is a `UserView` (`graph`) that paints the notes with `Pen`, with the `EnvelopeView` as a **transparent child on top** (`background_(Color.clear)`); `graph.onResize` keeps the envelope filling it. The backdrop shows the selected track's notes, or **all** notes for the combined track `"*"` (and for `tempo`, which is track-agnostic).
- **`lib/boardGui.scd`** — the **Xs-and-Os visualization board** (F2). A single compilation unit, `executeFile`'d by SETUP. Defines `~boardGui.([dimensions], [fullScreen])` (opens the board window — `dimensions` is a `Point` like `500@500`, the fixed board size drawn centered in the window however big it is; `fullScreen: true` covers the screen), `~boardEvent.(pitch, vel, dur)` (the hook the engine calls for every note on a `visual board` track), and `~toggleCueNum.()` (show/hide a current-cue-number overlay in the window's top-left corner — also the `q` key on the board).
- **`lib/synthTesterGui.scd`** — the **voice tester**. A single compilation unit, `executeFile`'d by SETUP. Defines `~openSynthTester` / `~synthGui`. Plays the engine's own cue SynthDefs (no separate server/defs) and lists exactly the `audio` targets in `~trackRoles` (the voices `tracks.txt` actually routes to), falling back to the known cue library if no track map is loaded. Queries each SynthDef's real controls via `SynthDescLib` and greys out sliders the selected voice doesn't declare. "play note" triggers a note of the current `dur` (heard through its self-freeing `dur`-timed ampEnv); "hold"/"release" plays a long note and drops `gate` (the fast killEnv) to audition the sustained body; "panic" frees the audition group. Distinct from the standalone `lib/synthOptions.scd` timbre-exploration lab, which boots its own server and defines its own non-engine synths.
- **Config files** (plain text, same family, all in **`data/`**): **`data/cues.txt`** the cue sheet (carving named beat-windows out of the `.mid` files in **`midi/`**; filenames in the sheet are bare basenames, resolved against `~midiDir` = `midi/`), **`data/tempos.txt`** the saved tempo curves, **`data/tracks.txt`** the track→synth/role map (global to the piece), **`data/midi_map.txt`** the saved MIDI bindings. All are created/edited by their respective save functions or by hand and auto-loaded by SETUP if present.

## Running

There is no build or test step — this is SuperCollider code evaluated interactively in the IDE (`scide`) or via `sclang`.

- **Dependency:** the `wslib` quark must be installed once: `Quarks.install("wslib");` then recompile the class library. `SimpleMIDIFile` (used to read MIDI) comes from wslib.
- **Boot + evaluate:** open `midi_cue_player.scd` (or `midi_cue_player_simple.scd`) and evaluate the single SETUP line (`this.executeFile("lib/setup.scd".resolveRelative)`) once with Ctrl+Enter. `lib/setup.scd` boots the server, loads the engine + GUIs, the track map, the cue sheet, the saved curves, and MIDI, then opens the board. Then trigger cues with the one-liners below it.
- **Performer commands** (after SETUP): `~play.(id)` takes a cue **name** (`"1"`, `"big finish"`, …) or, failing a name match, a **1-based** cue number (`~play.(1)` is the first cue, matching the score); `~next.()` plays the cue at the cursor **and advances** — tapping `~next`/`~play` while a cue is still sounding cuts it off and moves on to the next cue; `~stop.()` fades out the current cue and **stays on it** (so the next `~play`/`~next` restarts the cue you stopped), or — **if already stopped** — steps the cursor back one more cue without playing (to re-cue an earlier one).
- **MIDI learn** (after SETUP, which calls `~setupMIDI`): `~learnMIDINext.()` / `~learnMIDIStop.()` / `~learnMIDICue.(name)` arm a learn; the next incoming note-on/CC (value > 0) is bound to that action. Pass a path or `true` (e.g. `~learnMIDIStop.(true)`) to persist the whole map to the config file. `~learnMIDICancel.()` aborts.
- **Tempo GUI** (`~tempoGui.()`): opens the curve editor — pick a cue, drag breakpoints (x = beats, y = BPM), set a curve type per segment (named shape **or** a numeric curvature via the "or #:" box — 0 is straight, ± bends the segment), add/remove nodes (the `+ node`/`- node` buttons, **or `Ctrl-click` on the graph to add a node at the pointer and `Shift-click` to remove the nearest one** — endpoints stay pinned to `0..cueLen`), **audition** (plays the cue), **save all** (writes `tempos.txt`). Edits write straight into `~tempoEnvs[cueName]`, so `~play`/`~next` use them immediately; saving makes them persist.
- **Quick syntax/compile check without a server:** `sclang` one-liner — `thisProcess.interpreter.compile(File.readAllString("lib/cuePlayerEngine.scd".standardizePath)).notNil` returns `true` if the engine still compiles as one unit (catches the multi-region-block mistake).

## Architecture (one mental model to hold)

The core idea, and the reason the code is more than a MIDI player: **note times are kept in beats and never touched by the MIDI file's tempo map; tempo is re-imposed afterward from a `BPM-over-beats` envelope you write.** A second idea layers on top: **a cue can be a beat-window of a file, re-zeroed to its own beat 0**, so one MIDI file yields many cues.

**Windowing rule** (in `~readCueNotes`): a window is the half-open interval `[startBeat, endBeat)`. A note is kept only if its **onset** is inside the window — notes that started before `startBeat` are dropped (you don't hear a mid-note attack), notes whose onset is inside but that ring past `endBeat` are **truncated** to `endBeat`, and every kept note is shifted so the window's `startBeat` becomes beat 0.

The data flow, all via `~`-prefixed environment-variable functions defined in `cuePlayerEngine.scd`:

1. **Loading** — both loaders fill `~cues` and route note extraction through **`~readCueNotes.(path, startBeat, endBeat)`**, which reads a file with `SimpleMIDIFile` in default `\ticks` mode (beat = `ticks / division`, tempo discarded) and applies the window (see below). Two front doors:
   - **`~loadCues.(dir)`** — every `.mid`/`.midi` in `dir`, lexicographic sort, one whole-file cue each (`endBeat = inf`).
   - **`~loadCueSheet.(cuesTxtPath)`** — reads `cues.txt`, one cue per line `filename start_beat end_beat [optional name] [->]`. `#` comments and blank lines ignored; filenames are bare basenames resolved against `~midiDir` (the `midi/` folder) and must not contain spaces. This is the primary way to make several cues from one MIDI file. A trailing **`->`** token marks the cue **auto-follow**: when it finishes naturally it auto-triggers the next cue in the sheet (see Auto-follow below); it's stripped before the name field, so `... 13 ->` still names the cue `13`.

   Each cue is `(name:, file:, path:, startBeat:, endBeat:, autoNext:, notes:)`.

2. **`~tempoEnvs[key]`** — a `Dictionary` of `key` → an `Env` of **BPM over (window-relative) beats**. Lookup is via **`~tempoEnvFor.(cue)`**, which tries the cue's `name` first (specific window, e.g. `"tesCue.mid [10-20]"` or a custom 4th-column label), then the bare `file` basename (covers every window of that file), then `~defaultTempoEnv` (constant 120).

3. **`~buildTimeMap` / `~beatToTime`** — convert beats → seconds. `period(beat) = 60/(BPM(beat) · ~tempoMul)`; `time(beat)` is the cumulative-trapezoid integral of that period over a dense beat grid (`resolution = 0.01` beat). This integration is **agnostic to the Env's curve types** — whatever BPM shape you draw is rendered exactly. Querying past the envelope's duration clamps to (holds) the final BPM. **`~tempoMul`** is a global practice scale (default 1; `0.5` = half speed, `2` = double) applied here, so it speeds/slows notes *and* param curves together (they warp through the same map); it takes effect on the next `~play`.

4. **`~playCue.(index)`** — the primitive. Picks the cue's tempo env, maps every note's start/end beat through `~beatToTime`, resolves each note's **track → role/target** via `~trackRoleFor` (which SynthDef to play, or `\visual board`), sorts by absolute seconds, then plays them from a `Routine` on **`SystemClock` in absolute seconds** (no `TempoClock` — tempo is already baked in). `\audio` notes spawn their track's SynthDef; `\visual` notes are scheduled but currently no-op (the F2 board hook). Each cue gets its own `Group`. Interrupts whatever is playing first.

5. **Cursor model** — `~cueIndex` is a **cursor pointing at the cue `~next` will play next** (starts at 0). `~playCue.(i)` plays cue `i` then advances the cursor to `(i + 1)` (clamped at the last cue) **as the cue starts** — *play-then-advance* — so a direct `~play.(id)` also advances, and tapping `~next`/`~play` while a cue is still sounding cuts it off and moves on to the next. The single exception is `~stop`: when it cuts a sounding cue short it sets the cursor **back to that cue's index** (`~playing[\index]`), so `~next`/`~play` restarts it — *only `~stop` holds the cursor back*. `~stopCue` is the **internal primitive**: `\gate -> 0` on the group (fast fade via `killEnv`) then reaps it; it does **not** touch the cursor (the cursor reset lives in the `~stop` wrapper, so `~playCue`'s own interrupt of the previous cue still advances). `~playCue`'s Routine clears `~playing` and reaps the group on natural completion — so the system can tell "playing" from "finished/stopped" (what the two-mode `~stop` relies on).

6. **Performer wrappers** (bottom of the engine, the only names the Main file documents) — `~play` resolves a cue name to an index (`~cues.detectIndex` on `name`, else treats the arg as a 1-based cue number — `id - 1`) then calls `~playCue`; `~next` plays the cursor (`~playNext` → `~playCue.(~cueIndex)`); `~stop` is **two-mode**: if `~playing.notNil` it fades out via `~stopCue` and resets the cursor to the stopped cue's index (`~playing[\index]`), so the next `~play`/`~next` restarts it (this is the *only* path that holds the cursor back — an interrupt via `~play`/`~next` still advances), otherwise (already stopped) it steps the cursor back `~cueIndex - 1` **without playing**, to re-cue an earlier one. Everything else is internal.

7. **Window end, cursor advance, and auto-follow (cues.txt `->`)** — `~playCue` precomputes **`cueEndTime`** = the seconds from cue start to its **window end** (`endBeat - startBeat`, warped through the same tempo `map` as the notes; `nil` for an open-ended whole-file cue). The cue's `Routine` launches every note (its last `wait` is to the last note's *onset*) and then, crucially, **waits out the remainder to `cueEndTime` before finishing** — so the cue stays "playing" (`~playing` non-nil) through its trailing silence/release tail, and everything that marks the *end* of the cue happens at the **window boundary, not the last note**. This tail-wait lives *inside* the routine, so `~stopCue` (which stops the routine) **cancels it** — a stopped/interrupted cue never advances or auto-follows. At the natural window end the routine: clears `~playing`, **advances the overlay to the next cue** (`~boardCueIndex = ~cueIndex`, the play-then-advance cursor) and refreshes the board, and — **only if the cue is `autoNext`** — calls `~playNext` to chain into the next cue (skipped at the end of the sheet, `index + 1 < ~cues.size`). `~playing` is cleared first so `~playNext`'s `~stopCue` is a no-op and the finishing cue's release tails ring on into the next. So a plain cue advances only the *display* at its window end; a `->` cue advances display **and** audio there. The cursor needs no special handling: it already sits at `index + 1`.

### MIDI

`~setupMIDI` calls `MIDIClient.init` + `MIDIIn.connectAll` and installs two responders (`MIDIdef.noteOn`/`MIDIdef.cc`) that funnel every value-> 0 message into `~onMidi.(type, chan, num, val)`. `~onMidi` has two modes: if a learn is armed (`~midiLearning`), it records `"type chan num" -> action` into `~midiMap` (optionally saving); otherwise it looks the key up and calls `~fireAction`, which dispatches `\next`/`\stop`/cue-name. Bindings persist as a plain text file (`type chan num action`, e.g. `note 0 60 next`) via `~saveMidiMap`/`~loadMidiMap`; `~actionFromToken` maps `"next"`/`"stop"` back to symbols and treats anything else as a cue name. The default config path is `~midiMapPath`, set by SETUP to `data/midi_map.txt`; `~setupMIDI` auto-loads it if present.

### Tempo GUI & persistence

`tempoEnvGui.scd` edits the *same* `~tempoEnvs` dictionary the engine plays from. The bridge is two converters: **`~tgEnvToModel`** turns an `Env` into a `(beats:, bpms:, curves:)` model (absolute beats starting at 0, one curve per segment) and **`~tgModelToEnv`** turns it back (`Env(bpms, beatDurations, curves)`, BPM clamped ≥ 1). The editor uses a core **`EnvelopeView`**, which works in **normalized 0–1** on both axes and has *no* Env conversion of its own — so `refresh` maps model→view (`beat / maxBeat`, `(bpm - minBPM)/(maxBPM - minBPM)`) and the `action` callback (`readView`) maps view→model and writes `~tempoEnvs[name] = ~tgModelToEnv.(model)` on every drag. Hence changes are audible via `~play` immediately; **save all** is what persists them. Persistence is `~saveTempoEnvs`/`~loadTempoEnvs` with the line format `name | bpm levels | beat durations | curves` (the name is everything before the first `|`, so bracketed window names survive); `~parseCurve` reads a curve token as a numeric curvature if it's all `[0-9.eE+-]`, else a shape symbol. Default path `~tempoEnvPath` = `data/tempos.txt`, set and auto-loaded by SETUP (it overrides the inline `~tempoEnvs[...] = Env(...)` starting points).

The synths are a small library — `\sawCue`, `\pulseCue`, `\sineCue`, `\triCue`, `\velvetSawCue`, `\lorenzCue` — all sharing one arg contract (`out/freq/amp/dur/atk/rel/gate`, plus the F3 param controls `cutoff`/`pan`; `\velvetSawCue`/`\lorenzCue` swap `atk`/`cutoff` for their own `sharp`/`chaos`/`sat`) so any track can route to any of them. Each has two envelopes: `ampEnv` (**holds for the note's full `dur`, then releases over `rel`** — total life `dur + rel` — and self-frees via `doneAction: 2`, so the release tail rings *past* `dur` into the next note) and `killEnv` (a sustained gate that `~stopCue` drops to fade out cleanly without a click). They are stand-in timbres (distinct waveforms), with headroom for per-cue param controls (F3).

One additional voice, **`\metroCue`, is a sample player** rather than a synth: it plays the single sample `lib/sounds/metroblock.wav` **pitched by the note's `freq`** — playback rate `freq/440`, so A4 (A440) plays at 1×, and higher notes (C5/E5/…) play progressively faster/higher. Unlike the synth voices it **ignores `dur`** — it plays the whole (rate-scaled, stereo) sample once and self-frees (`PlayBuf doneAction: 2`); it still has the `killEnv` cue-stop fade. The buffer is loaded once at engine-load (`~metroBlockBuf`, since the engine is `executeFile`'d only after boot) and passed to each note by `~playCue` (the one target-specific branch in its otherwise-generic Synth creation). So `lib/sounds/` is now a runtime dependency for any track routed to `metroCue`.

**The note-duration contract is the SynthDef's job, not the routine's** (see gotchas): `~playCue`'s routine only spaces note *onsets* and never sends a per-note note-off, so every voice MUST bake its lifetime into a `dur`-timed `ampEnv` with `doneAction: 2`. A synth using a *gated/sustaining* env (e.g. `Env.adsr` held open by `gate`) without honoring `dur` will hang its notes until the cue's group is stopped — the original `\velvetSawCue` had exactly this bug.

### Tracks

`~readCueNotes` keeps each note's source `track:` index. `tracks.txt` maps `track role target [name...]` (role `audio` → a SynthDef name; role `visual` → `board`; the optional 4th+ tokens are a human-readable track **name**, stored as `~trackRoles[track][\name]` and read via `~trackName.(track)`, falling back to the index), loaded by SETUP via `~loadTrackMap` into `~trackRoles`; the mapping is **global to the piece**, not per-cue. `~trackRoleFor.(track)` looks a track up and returns **`nil` for any track not listed in `tracks.txt`** — `~playCue` filters those notes out, so an unmapped track is **discarded (silent)**, not routed to a default voice. (This means every track you want to hear must be listed; a marker/tempo track left out simply goes silent.) A cue plays all its mapped tracks simultaneously on its single tempo timeline, each track on its own voice.

### Visualization board (F2)

A track mapped `visual board` in `tracks.txt` feeds its notes (still on the cue's tempo-warped timeline, scheduled in the *same* `~playCue` routine) to **`~boardEvent.(pitch, vel, dur)`** in `boardGui.scd` (`dur` is the note's tempo-warped length in seconds). Pitch decodes to a move: **60–68** = Player 1 (X), **72–80** = Player 2 (O), `square = pitch - base` (0–8, row-major); **49** fades the **grid lines out** over `dur` and **47** fades the **grid lines in** (both drive `~boardGridFade` 0–1 via `~boardFadeGrid`, touching only the crisscross lines — not the marks or background; the grid **starts fully faded out**); **48** clears the board — **`~boardFadeOut` fades the marks out over `dur`** (the clear note's own length) on AppClock, then resets the state (so the clear note's duration is the fade time; a zero-length clear is instant). A new move cancels any in-flight fade (finishing the clear first, so it lands on a fresh board). **Chord exception:** a clear **chorded with a draw** (another visual note within `~clearChordDelta` ≈ 0.03 s) clears *instantly* instead of fading — `~playCue` flags it (`ev[\instantClear]`) and passes the board `dur 0`, and the event sort breaks ties so the clear (visual pitch 48) fires *before* the chorded draw, landing the new move on a freshly-cleared board. **Premature cutoff honors the clear instantly:** the engine tracks `~pendingClears` (clear notes in the playing cue not yet fired), and `~stopCue` — hit by `~stop` *and* by starting another cue — snaps the board to cleared (dur 0) if the interrupted cue had a clear that was still pending **or** mid-fade, so cutting a cue short never leaves a stale or half-faded board. **Velocity 127** (the board's win threshold) marks a winning-line cell — drawn **thick** and **coloured** (X vibrant red, O vibrant green); the newest move is also drawn thick. The board is a single **`UserView`** whose `drawFunc` (`~boardRender`) paints a real tic-tac-toe board with `Pen` — a white crisscross grid (no outer box, opacity `~boardGridFade`, starting hidden) and an X (two diagonals) / O (circle) stroked per occupied cell — so `~boardFade` and `~boardGridFade` are applied at draw time and `~boardDraw` just calls `~boardView.refresh`. Board state lives in `boardGui.scd` (`~boardState`/`~boardNewest`) and persists between cues — reset relies on the 48-clear note, so a window starting mid-game keeps prior state. The cue routine runs on `SystemClock`, so `~boardEvent` defers all GUI work to AppClock (`.defer`). It is a no-op if the board window isn't open, so audio doesn't depend on it. **Cue-number overlay:** `~toggleCueNum.()` (or the `q` key on the board) flips `~boardShowCueNum`; when on, `~boardRender` draws the 1-based number of **`~boardCueIndex`** in the window's top-left corner. `~boardCueIndex` is an explicit "currently shown cue" the engine maintains — set to the cue that **starts** in `~playCue` (the cue itself, *not* the play-then-advance cursor), to `~cueIndex` on idle cursor moves in `~announceCue`, and to `~cueIndex` at a cue's **natural window end** inside the play routine (see Window end above). Keeping it explicit (rather than deriving from `~playing`/`~cueIndex`) is deliberate: it holds steady through the playing cue's trailing silence/release tail and only advances at the true window boundary — so the number switches when the **window** finishes, not when the last note ends. Because that window-end advance lives inside the routine, a `~stop`/interrupt cancels it (the routine is stopped), leaving the display on the stopped cue. The engine also refreshes the board (deferred, only when the overlay is on) at each transition, so the number tracks `~play`/`~next`/`~stop`/arrow moves and auto-follow handoffs live.

### Parameter curves (F3)

Beyond tempo, any `(cue, track, param)` can have its own **value-over-beats** `Env` in **`~paramEnvs`** (keyed by the composite string `"cue\ttrack\tparam"`; track `"*"` = all tracks of the cue). `~paramList` (`\cutoff`, `\pan`, `\amp`, …) names the editable params — each must be a control the SynthDefs declare (the shared synth contract includes `cutoff` + `pan`). Control model is **continuous control buses** (not onset sampling): in `~playCue`, **`~paramCurvesFor.(cueName)`** gathers the cue's curves and, for each `(track, param)`, allocates a `Bus.control` and plays a **`\paramBus`** writer synth in the cue's group. The writer is fed a **time-domain** `Env` produced by **`~warpParamEnv`**, which densely samples the beat-domain curve and warps each beat through the same tempo `map` the notes use (so the drawn shape is rendered faithfully under the tempo curve — mirroring `~buildTimeMap`); it has no `doneAction`, so it **holds its final value** until the group is freed. Each audio note then `.map`s its matching controls to those buses (resolved per track: a track-specific curve overrides the `"*"` curve for the same param), so params **glide continuously, including during held notes**. **The `amp` curve is special-cased:** it drives a separate `ampScale` control (default 1) that *multiplies* the velocity-derived `amp`, so an amp curve **scales** velocity rather than replacing it (every synth declares `ampScale`). The param→control mapping lives in **`~paramControlName`** (`amp`→`ampScale`, else identity), used both by `~playCue` and the introspection helpers. **A freshly-created curve seeds flat on the SynthDef's declared default** for that control: `~paramCurveDefault.(cue, track, param)` resolves the track's synth (for `"*"`, the cue's first audio synth declaring the control) and reads the default via **`~synthControlDefault`** (which queries `SynthDescLib.global`), falling back to the param's `~paramRanges` low if unknown — so an untouched curve is a no-op vs. not drawing one. Each entry in `~paramRanges` (the editor y-axis range) must therefore contain that control's default (e.g. `cutoff` is `[20, 20000]` because the synths default `cutoff` to 20000). Buses are freed alongside the group on stop/interrupt and on natural completion. Tempo stays the privileged exception — per-cue, in `~tempoEnvs`, never per-track. Persistence is unified into `tempos.txt`: the line format is `name | track | param | levels | durations | curves` (tempo written as `name | * | tempo | …`); `~loadTempoEnvs` still reads the legacy 4-field tempo lines (`name | levels | durations | curves`).

## Conventions / gotchas specific to this code

- **Curve semantics matter and are intentional.** In a BPM envelope, `\lin` is *not* a linear-in-period accel and `\exp` is geometric-in-BPM. Use `\exp` for a smooth musical accelerando. The engine renders any curve faithfully, so the choice is purely musical.
- **Tempo envelopes are keyed by cue name (or file basename as fallback).** For a window the name is `"<file> [<start>-<end>]"` using the *text as written in `cues.txt`* (so `0 10` → `"tesCue.mid [0-10]"`, not `[0.0-10.0]`), or the 4th-column label if you gave one. The number list is BPM values; the durations list is in **window-relative beats**.
- A tempo env shorter than the cue is fine — it sustains its last BPM. A cue with no entry plays at a flat 120.
- **Note duration is the SynthDef's responsibility.** `~playCue`'s routine spaces note *onsets* only and never sends a per-note note-off, so a voice must bake its lifetime into a `dur`-timed `ampEnv` (`doneAction: 2`). A gated/sustaining env not driven by `dur` will hold every note until the cue's group is stopped (the original `\velvetSawCue` bug).
- **Unmapped tracks are discarded.** `~trackRoleFor` returns `nil` for tracks not in `tracks.txt`, and `~playCue` drops their notes — there is no default voice, so a track you forget to list goes silent.
- Paths use SuperCollider idioms: `"data/cues.txt".resolveRelative` (relative to the `.scd` file), `.standardizePath`, `.pathMatch`.
